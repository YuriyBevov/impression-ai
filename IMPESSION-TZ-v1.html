1. Общая информация о проекте

1.1. Назначение системы

Система Impression предназначена для управления процессом ИИ-перевода, клиентскими данными, терминологическими базами знаний и финансовой аналитикой затрат на ИИ-модели.

Веб-интерфейс должен стать единой рабочей панелью для сотрудников бюро переводов, где доступны:

  • управление клиентами;
  • управление базой знаний и терминологическими коллекциями;
  • запуск переводов через интеграцию с n8n;
  • контроль и анализ затрат на ИИ-провайдеров;
  • работа с пользовательскими и общими словарями/коллекциями.

1.2. Бизнес-цель

Создать внутреннюю систему, которая:

  • ускоряет переводческие процессы;
  • повышает консистентность терминологии;
  • обеспечивает клиент-специфичный стиль перевода;
  • снижает ручную операционную нагрузку;
  • позволяет контролировать расходы на использование ИИ.

1.3. Основные пользователи системы

Роль Описание Основные действия
Администратор Полный доступ ко всем сущностям и настройкам управление пользователями, клиентами, коллекциями, бюджетом
Менеджер/оператор Работает с переводами, клиентами, базой знаний запуск переводов, работа с документами, правка записей
Аналитик/руководитель Просматривает расходы и отчёты анализ бюджета, сравнение стоимости переводов
Переводчик/редактор Проверяет и редактирует результаты просмотр результатов, уточнение терминов, работа с KB

Примечание: на текущем этапе роли можно реализовать как минимум в двух уровнях:
Admin и User. Расширенная ролевая модель может быть отложена на V2/V3.

2. Цели и задачи разработки

2.1. Цели MVP

Реализовать минимально достаточную версию системы, которая позволит:

  1. авторизоваться в системе (логин/пароль, JWT);
  2. управлять клиентами (PostgreSQL);
  3. запускать перевод текста через n8n;
  4. работать с коллекциями Qdrant и их записями;
  5. видеть базовую финансовую статистику по затратам на ИИ (OpenRouter API).

2.2. Задачи системы

  • централизовать работу с терминологией;
  • привязать терминологию к конкретным клиентам;
  • обеспечить приоритетную подстановку знаний при переводе;
  • упростить пополнение базы знаний через загрузку документов;
  • обеспечить прозрачность стоимости переводов и расходов на ИИ.

3. Границы проекта

3.1. Входит в объём работ

  • веб-интерфейс (Vue 3 + PrimeVue);
  • авторизация (JWT, bcrypt);
  • CRUD для клиентов (PostgreSQL);
  • CRUD для коллекций и записей в Qdrant;
  • поиск по коллекциям и записям;
  • запуск перевода текста через n8n;
  • предпросмотр и ручная корректировка записей перед сохранением;
  • бюджетная аналитика по OpenRouter API;
  • базовая дедупликация записей;
  • история переводов (по клиенту + по каждому переводу).

3.2. Не входит в MVP

  • полноценная OCR-обработка изображений;
  • массовая обработка больших пакетов документов;
  • сложная RBAC-модель с гибкими правами;
  • мультиязычный интерфейс;
  • редактор сложных workflow внутри UI;
  • автоматическое выставление счетов клиентам;
  • CRM-функции вне базового учёта клиентов.

5. Функциональные требования

5.1. Авторизация и доступ

5.1.1. Вход в систему

Система должна предоставлять страницу авторизации с полями:

  • логин;
  • пароль;
  • кнопка «Войти».

Базовые требования

  • аутентификация по логину и паролю;
  • bcrypt-хэширование паролей;
  • сессии через JWT-токены;
  • защита маршрутов от неавторизованного доступа (router guards);
  • отображение имени пользователя в шапке;
  • возможность выхода из системы.

5.1.2. Создание пользователей

  • Администратор создаёт пользователей вручную (через интерфейс или напрямую в БД);
  • Без пригласительных токенов / регистрации.

5.1.3. Выход из системы

Кнопка «Выйти» должна быть доступна:

  • в шапке;
  • в сайдбаре.

При выходе: сессия завершается, редирект на страницу логина.

5.2. Раздел «База знаний»

5.2.1. Назначение раздела

Раздел предназначен для управления коллекциями и записями в Qdrant, а также для загрузки документов с последующим извлечением терминологических пар.

5.2.2. Основные сущности

Сущность Описание
Коллекция Логическая группа записей в Qdrant
Запись / Point Единица данных внутри коллекции
Клиентская коллекция Коллекция, привязанная к конкретному клиенту (client_{client_id})
Общая коллекция Глобальный glossary
KB коллекция Дополнительная база знаний общего назначения

5.2.3. Сценарии работы

A. Просмотр списка коллекций

  • видеть список всех коллекций;
  • искать коллекции по названию;
  • фильтровать по типу (клиентская / общая / KB);
  • видеть привязку к клиенту.

B. Переход внутрь коллекции

  • название, тип, связанный клиент, количество записей;
  • поиск по ключу/значению;
  • список записей с пагинацией.

C. CRUD для записей

  • создание новой записи;
  • редактирование записи;
  • удаление записи;
  • дополнение записи новыми полями/метаданными.

5.2.4. Структура записи (Qdrant)

Поле Тип Обязательное Описание
id string / uuid да внутренний ID
source_term string да исходный термин/фраза
target_term string да перевод / целевое значение
source_lang string нет язык исходного термина
target_lang string нет язык перевода
context text нет контекст использования
note text нет комментарий редактора
tags array нет теги
client_id uuid нет привязка к клиенту
collection_type enum да client / glossary / kb
created_at datetime да
updated_at datetime да
created_by user_id нет автор записи

5.2.5. Поиск

  1. Поиск по коллекциям — название, тип, клиент
  2. Поиск по записям во всех коллекциях — полнотекстовый по ключам/значениям, фильтр по клиенту/типу
  3. Поиск внутри коллекции — по ключу, значению, тегам, контексту

5.2.6. Загрузка документов

Сценарий

  1. Выбор коллекции → загрузка файла
  2. ИИ-обработка (Qwen 3 Coder Plus)
  3. Извлечение терминологических пар
  4. Предпросмотр → ручное редактирование → подтверждение → сохранение

Форматы (MVP)

  • .txt, .docx, .pdf
  • Пакетная загрузка: да

Форматы (будущее)

  • .xlsx, .csv
  • изображения через OCR

5.2.7. Дедупликация

  • MVP: поиск полных совпадений source_term + target_term
  • UX: предупреждение, выбор: пропустить / заменить / объединить / сохранить как новую

5.3. Раздел «Сделать перевод»

5.3.1. Назначение

Запуск переводов через n8n.

5.3.2. Основной сценарий MVP

  1. Выбор клиента (из выпадающего списка)
  2. Выбор направления: RU→EN или EN→RU
  3. Вставка текста в поле ввода
  4. Нажатие «Перевести»
  5. Отправка в n8n
  6. Получение и отображение результата

5.3.3. Элементы страницы

Элемент Описание
Выбор клиента выпадающий список (обязательно)
Направление перевода RU→EN / EN→RU
Текст ввода большое текстовое поле
Кнопка «Перевести» запуск
Блок результата перевод + метаданные
История запросов список предыдущих переводов (по клиенту + по каждому)

5.3.4. Обязательная логика

  • перевод без выбора клиента невозможен
  • при выборе клиента применяется его коллекция как приоритетная
  • приоритет: Client → Glossary → KB

5.3.5. История переводов

  • Да, нужна в MVP
  • Группировка по клиентам
  • Детализация по каждому переводу (дата, модель, стоимость, статус)

5.4. Раздел «Клиенты»

5.4.1. Назначение

Ведение клиентской базы. Данные хранятся в PostgreSQL.

5.4.2. Сценарии

A. Создание клиента

  • Администратор заполняет форму
  • Автоматическое создание коллекции в Qdrant client_{client_id}

B. Редактирование

  • Изменение любых полей, статуса (активен / архив)

C. Архивация

  • Клиент скрывается из выпадающих списков
  • Данные и коллекция не удаляются

5.4.3. Структура клиента (PostgreSQL)

Поле Тип Обязательное Описание
id uuid да внутренний ID
name string да название / имя клиента
inn string нет ИНН
legal_address string нет юридический адрес
email string нет email
phone string нет телефон
contact_info text нет доп. комментарий по контакту
notes text нет внутренние заметки
status enum да active / archived
collection_id string да название коллекции в Qdrant
created_at datetime да
updated_at datetime да

5.5. Раздел «Бюджет»

5.5.1. Назначение

Финансовая аналитика затрат на ИИ-модели.

Источник данных: OpenRouter API

5.5.2. Сценарии

  1. Общий бюджет за период
  2. Разбивка по клиентам
  3. Разбивка по моделям
  4. Детализация по каждому переводу (стоимость, модель, токены)

5.5.3. Визуализация

MVP: таблица затрат + базовые метрики V2/V3: графики, экспорт, прогнозирование

6. Хранение данных (архитектура)

6.1. Две системы хранения

PostgreSQL (реляционные данные)

  • Таблица users — учётные записи пользователей (логин, bcrypt-хэш пароля, роль)
  • Таблица clients — реляционные данные клиентов

Qdrant (векторные/терминологические данные)

  • Коллекции glossary/kb — терминологические пары с векторами
  • Коллекция client_{client_id} — персональная коллекция клиента

6.2. Иерархия коллекций

1. Пользовательский запрос на перевод
   │
   ▼
2. Client Collection (client_{id}) — наивысший приоритет
   │ Есть совпадение? → используем
   │ Нет → падаем ниже
   ▼
3. Glossary Collection
   │ Есть совпадение? → используем
   │ Нет → падаем ниже
   ▼
4. KB Collection
   │ Есть совпадение? → используем
   │ Нет → стандартный перевод модели

6.3. Влияние на UI

  • Создание клиента → создаётся коллекция в Qdrant
  • База знаний → отображение типа коллекции и связанного клиента
  • Перевод → выбранный клиент определяет приоритетную коллекцию

8. Интеграции

8.1. n8n

  • UI → n8n webhook (POST)
  • Параметры: client_id, source_text, source_lang / target_lang
  • Ответ: translated_text, status, model, cost, tokens

8.2. Qdrant

  • UI → nginx proxy → Qdrant REST API
  • CRUD через /api/qdrant/

8.3. OpenRouter

  • UI → OpenRouter API (чтение баланса, истории)
  • Отображение в разделе «Бюджет»
